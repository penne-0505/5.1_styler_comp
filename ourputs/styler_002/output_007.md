件名：来週（11/17）からの新しいコードレビュー運用ルールについて

各位

いつもお世話になっております。
エンジニアの〇〇です。

来週 11/17（月）より、プロダクト開発におけるコードレビューの運用ルールを一部変更・統一いたしますのでご案内いたします。

## 1. 背景

現在、コードレビューは各チーム・各メンバーの裁量に依存しており、

* レビューの深さや観点にばらつきがある
* レビュー依頼から対応までの時間が読みづらい
* ノウハウや設計意図が個人に閉じてしまう

といった課題が見られます。

これらを解消し、組織として一貫した品質基準を持つために、コードレビューの運用ルールを整理し直すことにしました。

## 2. 目的

今回のルール変更の主な目的は以下のとおりです。

* 変更の影響範囲やリスクを、事前にレビューで検知しやすくする
* レビューを通じて知見・設計意図・ベストプラクティスを共有し、属人化を減らす
* 開発スピードと品質のバランスを取りやすくする

開発メンバーだけでなく、プロダクトに関わる全メンバーに関係する取り組みのため、共有させていただきます。

## 3. 新しいコードレビュー運用ルール

対象: 11/17（月）以降に作成される、メインブランチへのマージを前提としたすべてのプルリクエスト（PR）

### 3-1. レビュー必須条件

1. **メインブランチへのマージ前に、原則として他者レビューを必須**とします。
2. **自己承認（自分で作成したPRを自分で承認してマージ）は禁止**とします。
3. 緊急対応など、例外的にこの限りでない場合は、事後でも必ずレビューを依頼してください。

### 3-2. レビュー体制とレスポンス

1. 各PRには、**少なくとも1名のレビュワー**をアサインしてください（可能な限り2名以上を推奨）。
2. レビュワーは、**レビュー依頼から1営業日以内に一次リアクション（コメントまたはLGTM/承認）**を行うことを目安とします。
3. レビュワーの選定は、モジュールの担当者・関連ドメインの知識を持つメンバーを優先してください。

### 3-3. PR作成時の記載内容

PR 作成時には、テンプレートに沿って、以下を必ず明記してください。

* 変更概要（What）
* 目的・背景（Why / Issueリンクなど）
* 影響範囲（関連する機能・モジュール・他サービスなど）
* 動作確認内容（実施したテスト・確認手順・結果）
* レビューしてほしい観点（例：設計、パフォーマンス、例外処理、UIなど）

これにより、レビュワーが短時間で状況を把握できるようにします。

### 3-4. レビュー観点とコメント対応

1. レビュワーは、最低限以下の観点で確認してください。

   * 仕様どおりの動作になっているか
   * 設計として妥当か（責務分離、拡張性、保守性など）
   * コードの可読性（命名、コメント、構造）
   * テストの有無・内容が妥当か
2. PR 作成者は、コメントへの対応方針（修正する／しない理由）をコメントで明示し、対応完了後はスレッドを解決状態にしてください。

### 3-5. マージ条件

1. 必要な人数の承認（Approve/LGTM）が付いた PR のみ、メインブランチへのマージを行ってください。
2. 未解決コメントが残っている場合、マージは原則として行わないでください（緊急時は、事前にレビュワーと合意のうえで対応）。

## 4. メンバーの皆さまへのお願い

* 開発メンバーの方へ

  * 上記ルールに沿った PR 作成とレビュー対応をお願いします。
  * 実際に運用する中で不都合や改善案があれば、遠慮なくフィードバックをお寄せください。

* 開発以外のメンバーの方へ

  * 開発スケジュールにおいて、コードレビューの時間を前提とした見積もり・調整が必要になる場合があります。
  * 今回の取り組みは、プロダクト品質向上とトラブル削減につながるものですので、ご理解いただけますと幸いです。

以上となります。
本ルールはまずは 11/17（月）から運用開始し、一定期間運用したうえで内容を見直す予定です。

引き続き、より良い開発プロセスの構築にご協力をお願いいたします。
